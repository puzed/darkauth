# DarkAuth User Key Management

Status: normative target
Version: v2

This document specifies DarkAuth key management for password auth, upstream federation, SCIM-provisioned accounts, passkeys, trusted devices, recovery, and zero-knowledge OAuth clients.

The design separates identity authentication from encryption-key unlock:

- Authentication establishes the DarkAuth subject and policy context.
- Key unlock makes account encryption material available to the current browser/device.

DarkAuth MUST preserve the zero-knowledge property that server-side code and server-side storage never receive plaintext user root keys, app keys, wrapping keys, passkey PRF outputs, recovery keys, or decrypted user data during honest operation.

## 1. Goals

- Separate login/authentication from key unlock.
- Support local password login, federated SSO, passkeys, and future auth methods without forcing all of them to derive encryption keys directly.
- Support SCIM for enterprise provisioning and deprovisioning.
- Deliver a unique key to each OAuth client instead of delivering the same user root key to every ZK-enabled client.
- Allow users to approve new devices from an existing unlocked device.
- Allow users to manage passwords, passkeys, trusted devices, recovery keys, and key envelopes in the user UI.
- Avoid storing plaintext root/app keys in cookies, localStorage, sessionStorage, IndexedDB, or server storage.
- Preserve OIDC compatibility for clients that do not request zero-knowledge key delivery.
- Keep fragment-only encrypted key delivery and token-response hash binding for hosted web clients.

## 2. Non-Goals

- SCIM is not an authentication protocol. DarkAuth will implement SCIM as provisioning. A SCIM-provisioned user authenticates with a configured auth method such as DarkAuth password, passkey, or enterprise SSO.
- Passkeys without PRF do not unlock encrypted data by themselves. They can authenticate the user, but a separate key unlock step is required.
- Browser-hosted zero-knowledge cannot protect against malicious JavaScript served by trusted origins, XSS, malicious extensions, or a compromised device.
- The server must never escrow a plaintext recovery key or plaintext account root key.
- This spec does not require SAML before OIDC federation. SAML is included as an enterprise federation option.

## 3. Terminology

- Account Root Key (ARK): A random 32-byte user root key generated client-side. This is the canonical user encryption root.
- Client App Key (CAK): A per-OAuth-client key derived from the ARK. ZK clients receive the CAK, not the ARK.
- Key Envelope: A server-stored ciphertext that wraps the ARK for one unlock method.
- Unlock Method: A method that can unwrap or receive the ARK locally, such as password, passkey PRF, trusted device, or recovery key.
- Auth Method: A method that proves identity, such as password login, federated SSO, passkey authentication, or an existing DarkAuth session.
- Trusted Device: A browser/device with local key material that can unwrap the ARK or approve another device.
- Keybag: The set of account keys and envelopes associated with a user.
- ZK Client: An OAuth client registered to request client-side key delivery.
- Legacy DRK: The v1 per-user Data Root Key. During migration, an existing DRK is treated as the initial ARK for that user.

## 4. Core Security Model

- The ARK MUST be generated with cryptographically secure randomness in the browser or another trusted client runtime.
- The ARK MUST NOT be generated by the server.
- The ARK MUST NOT be delivered to OAuth clients.
- ZK clients MUST receive only a CAK derived for their own `client_id` and requested context.
- The server MAY store key envelopes, metadata, hashes, public keys, and audit events.
- The server MUST NOT store plaintext ARK, CAK, password-derived wrapping keys, passkey PRF outputs, recovery secrets, or device private keys.
- Auth methods and unlock methods MUST be modeled separately in data, API, UI, and audit logs.

### 4.1 Identity State Versus Key State

A user session has two independent states:

```
identity_state = anonymous | authenticated | mfa_pending | suspended
key_state      = none | locked | unlocked | setup_required | recovery_required
```

Rules:

- `authenticated + locked` is valid. It means DarkAuth knows who the user is, but the browser has not unlocked ARK.
- `authenticated + setup_required` is valid for SCIM-created, SSO-created, admin-created, or passkey-only users with no ARK yet.
- Non-ZK OAuth clients only require `identity_state = authenticated` and normal policy checks.
- ZK OAuth clients require `identity_state = authenticated` and either `key_state = unlocked` or a successful unlock/setup journey during authorization.
- Admin and user UI MUST avoid wording that says a user is fully signed in to encrypted apps while `key_state != unlocked`.

### 4.2 Auth Method Versus Unlock Method Matrix

| Method | Proves identity | Can unlock ARK | Notes |
| --- | --- | --- | --- |
| OPAQUE password | Yes | Yes | `export_key` derives password envelope wrapping key. |
| Upstream OIDC/SAML SSO | Yes | No | Requires separate unlock for ZK clients. |
| SCIM | No | No | Provisioning/lifecycle only. |
| WebAuthn passkey without PRF | Yes | No | Requires separate unlock for ZK clients. |
| WebAuthn passkey with verified PRF | Yes | Yes | Only after PRF envelope setup. |
| Trusted device approval | No by itself | Yes | Requires an authenticated target session plus approval by an existing unlocked device. |
| Recovery key | No by itself | Yes | Requires authenticated or recovery-verified account context. |

## 5. Key Hierarchy

### 5.1 Account Root Key

The ARK is the stable account root for the user.

```
ARK = random(32 bytes)
```

The ARK is stored server-side only as encrypted envelopes.

### 5.2 Client App Key Derivation

For a ZK OAuth client, the user UI derives a client-specific key:

```
CAK = HKDF-SHA256(
  ikm = ARK,
  salt = H("DarkAuth|v2|client-key|sub=" + sub + "|key_id=" + key_id),
  info = "client_id=" + client_id + "|org_id=" + org_id + "|aud=" + aud,
  length = 32
)
```

Rules:

- `client_id` MUST be included.
- `sub` and `key_id` MUST be included.
- `org_id` MUST be included when the authorization is organization-scoped.
- `aud` SHOULD be included when distinct from `client_id`.
- The derivation version MUST be explicit.
- Existing v1 clients MUST NOT silently switch from DRK/ARK delivery to CAK delivery without a migration flag, because existing ciphertext may depend on the old key.

### 5.2.1 CAK Context And Stability

The same user, same `client_id`, same `key_id`, same derivation version, and same organization context MUST produce the same CAK.

Different `client_id` values MUST produce different CAKs.

Organization-scoped apps have two supported policies:

- Account-wide client key: omit `org_id` from derivation and use the same CAK across organizations.
- Organization-scoped client key: include `org_id` and produce different CAKs per organization.

The client registration MUST choose one policy. The default SHOULD be organization-scoped when the client stores organization-partitioned encrypted data.

### 5.2.2 Root Versus App Key Delivery

v2 ZK delivery MUST NOT deliver ARK to OAuth clients.

DarkAuth user UI may hold ARK in memory long enough to derive CAK and manage envelopes. RP apps should receive only CAK or app-specific wrapped data keys.

### 5.3 Password Envelope

DarkAuth password login continues to use OPAQUE.

```
MK_password = HKDF-SHA256(export_key, salt = H("DarkAuth|v2|password|sub=" + sub), info = "mk")
KW_password = HKDF-SHA256(MK_password, salt = "DarkAuth|v2", info = "account-root-wrap")
password_envelope = AEAD_Encrypt(KW_password, ARK, aad = envelope_aad)
```

The OPAQUE `export_key` MUST remain client-side.

### 5.3.1 Password Change

Password change MUST preserve ARK when the old password is known:

1. Verify old password with OPAQUE step-up.
2. Unwrap ARK using the old password envelope.
3. Register/update the new OPAQUE record.
4. Derive the new password wrapping key from the new `export_key`.
5. Create a new password envelope for the same ARK.
6. Revoke or archive the old password envelope according to policy.

If the old password is not known, email reset restores account access only. It MUST NOT decrypt existing ARK envelopes. The user must use recovery key, PRF passkey, trusted device approval, or generate a new ARK and accept loss of old encrypted app data.

### 5.4 Passkey PRF Envelope

When WebAuthn PRF is supported by the browser and authenticator, a passkey can be both an auth method and an unlock method.

```
prf_output = WebAuthn.get(... extensions.prf.eval ...)
KW_passkey = HKDF-SHA256(prf_output, salt = envelope_salt, info = "DarkAuth|v2|passkey-prf|credential_id=...")
passkey_envelope = AEAD_Encrypt(KW_passkey, ARK, aad = envelope_aad)
```

Rules:

- Passkeys without PRF MAY be used for authentication only.
- A passkey MUST NOT be shown as a key-unlock method unless a real PRF result was returned and verified during setup.
- The server stores credential public keys, PRF salt metadata, and wrapped ARK envelopes, but never PRF output.

### 5.5 Trusted Device Envelope

A trusted device SHOULD use a local non-extractable wrapping key when available.

Preferred browser profile:

- Generate a non-extractable WebCrypto AES-GCM key or ECDH private key.
- Store the `CryptoKey` in IndexedDB if supported by the browser.
- Store a server-side envelope that wraps the ARK for that device.
- If IndexedDB `CryptoKey` persistence is unavailable or unreliable, fall back to memory-only device approval.

Important limitation:

- A non-extractable key reduces at-rest exfiltration risk, but XSS running on the trusted origin can still ask the browser to decrypt while the key is usable.

### 5.6 Recovery Key Envelope

Users SHOULD be able to create an offline recovery key.

Rules:

- Recovery keys MUST be high entropy.
- Recovery keys MUST be shown once and never stored plaintext server-side.
- The recovery key SHOULD wrap the ARK through a memory-hard KDF or HKDF over high-entropy input.
- Recovery use SHOULD rotate or rewrap the ARK and record a security event.

### 5.7 Envelope Format

Every envelope MUST have explicit metadata and authenticated data.

Envelope plaintext:

```json
{
  "typ": "DarkAuth-Account-Root-Key",
  "version": "v2",
  "sub": "<user-sub>",
  "key_id": "<account-key-id>",
  "ark": "<base64url 32 bytes>",
  "created_at": "<iso8601>"
}
```

Envelope AAD:

```text
DarkAuth|v2|envelope
sub=<sub>
key_id=<key_id>
envelope_id=<envelope_id>
type=<password|passkey_prf|trusted_device|recovery>
wrapping_alg=<alg>
```

Rules:

- AAD MUST include `sub`, `key_id`, `envelope_id`, `type`, and `wrapping_alg`.
- Envelope decrypt MUST verify plaintext `sub` and `key_id` match server metadata.
- Envelope IDs MUST be random, unguessable identifiers.
- Envelope metadata may be visible to admins, but ciphertext and secrets are never decryptable server-side.

## 6. Browser Storage Rules

- Plaintext ARK and CAK MUST be memory-only by default.
- Plaintext ARK and CAK MUST NOT be stored in localStorage.
- Plaintext ARK and CAK MUST NOT be stored in JS-readable cookies.
- Plaintext ARK and CAK SHOULD NOT be stored in sessionStorage. Session-scoped storage may hold OAuth state, PKCE verifier, and ephemeral private keys only when callback continuity requires it.
- HttpOnly cookies are appropriate for sessions and refresh credentials, but not for browser-side decryption keys because JavaScript cannot read HttpOnly cookies.
- Persistent "remember this device" MUST store only encrypted envelopes and local non-extractable key handles where possible.
- If the browser loses local key material, the user must unlock again with password, passkey PRF, recovery key, or existing-device approval.

### 6.1 Remembered Browser Profile

Browsers do not provide a universal secure enclave for web apps. The best available web profile is:

1. Generate a non-extractable WebCrypto key.
2. Store the `CryptoKey` handle in IndexedDB where supported.
3. Store an ARK envelope encrypted for that local key on the server or in IndexedDB.
4. Require user reauthentication or device approval if the key handle is lost.

This protects better against at-rest storage disclosure than localStorage plaintext, but it does not protect against active same-origin JavaScript that can call decrypt while the browser profile is unlocked.

Cookies:

- HttpOnly cookies are for auth/session transport.
- JS-readable cookies MUST NOT contain ARK, CAK, PRF output, recovery keys, or password-derived keys.
- Cookies SHOULD NOT be used for key envelopes unless there is a specific same-site transport reason and size constraints are acceptable.

## 7. ZK Client Key Delivery

DarkAuth keeps the current fragment JWE idea, but changes the payload from account root material to client-scoped material.

### 7.1 Authorization Request

The RP SDK generates an ephemeral ECDH keypair and sends the public key:

```
/authorize?...&zk_pub=base64url(JSON.stringify(public_jwk))
```

Rules:

- The private key MUST stay in the RP browser.
- The private key SHOULD be non-extractable where callback continuity allows.
- If the private key must survive redirect, it MAY be stored temporarily by OAuth `state` and MUST be deleted after callback.
- The server MUST validate `zk_pub` format and curve membership.
- The server MUST NOT log `zk_pub`.

### 7.2 User UI Key Delivery

After authentication and key unlock, the DarkAuth user UI:

1. Unwraps ARK locally.
2. Derives CAK for the requesting `client_id`.
3. Encrypts CAK to the RP ephemeral `zk_pub` as compact JWE.
4. Computes `cak_jwe_hash = base64url(SHA-256(cak_jwe))`.
5. Sends only the hash to `/authorize/finalize`.
6. Redirects to the RP with the JWE in the URL fragment.

The JWE payload MUST include:

- `typ`: `DarkAuth-Client-Key`
- `version`: key delivery version
- `sub`
- `client_id`
- `aud`
- `org_id` when present
- `request_id`
- `state_hash`
- `redirect_uri_hash`
- `key_id`
- `key_kind`: `client_app_key`
- `cak`: base64url CAK
- `iat`
- `exp`

The RP SDK MUST verify metadata before accepting the key.

### 7.2.1 Fragment And Token Field Names

New v2 clients use these fields:

- Fragment parameter: `darkauth_key_jwe`
- Token response hash: `zk_key_hash`
- Token response kind: `zk_key_kind = "client_app_key"`
- Token response version: `zk_key_version = "v2"`

Legacy clients use these fields:

- Fragment parameter: `drk_jwe`
- Token response hash: `zk_drk_hash`
- Payload kind: legacy DRK

Client registration MUST select the key delivery version:

```
key_delivery_version = "v1-drk" | "v2-client-key"
```

DarkAuth MUST NOT infer the version from client behavior when a client is already registered.

### 7.2.2 RP Verification Requirements

The RP SDK MUST reject the delivered key unless all checks pass:

- JWE hash matches the token response hash.
- `typ` is `DarkAuth-Client-Key`.
- `key_kind` is `client_app_key`.
- `client_id` matches the configured client.
- `sub` matches the ID token subject.
- `aud` matches the expected audience.
- `state_hash` matches the OAuth state sent by the SDK.
- `redirect_uri_hash` matches the redirect URI used for code exchange.
- `exp` is in the future and within the allowed short lifetime.
- `key_id` is present.

On failure, the SDK MUST clear callback state and MUST NOT expose the key to application code.

### 7.3 Token Endpoint Binding

The token response for a ZK authorization includes:

```
zk_key_hash = base64url(SHA-256(fragment_jwe))
zk_key_kind = "client_app_key"
zk_key_version = "v2"
```

Rules:

- The token endpoint MUST NOT return the JWE.
- The server stores only the hash, not the JWE.
- The RP MUST verify the fragment JWE hash before decrypting.
- The RP MUST remove the fragment from browser history after handling.

### 7.4 Non-ZK Clients

When a client does not request ZK key delivery:

- DarkAuth MUST NOT force ARK setup or unlock unless organization policy explicitly requires key setup for all sign-ins.
- `/token` MUST omit key hash fields.
- The client uses normal OIDC behavior.

## 8. Login And Unlock Journeys

### 8.1 Password Login With ZK Client

- User authenticates using OPAQUE.
- Browser receives OPAQUE `export_key`.
- Browser derives password wrapping key.
- Browser unwraps ARK from password envelope.
- Browser derives CAK for the requesting client.
- Browser sends encrypted CAK by fragment JWE.

### 8.2 Password Login Without ZK Client

- User authenticates using OPAQUE.
- No key unlock is required unless the user UI itself needs key management.
- DarkAuth issues normal OIDC tokens.

### 8.3 Federated SSO With ZK Client

- User authenticates through an upstream IdP.
- DarkAuth maps the upstream identity to a local `sub`.
- If the client does not request ZK delivery, issue normal OIDC tokens.
- If the client requests ZK delivery, show key unlock choices:
  - Accept on another device.
  - Enter DarkAuth password.
  - Use a PRF-capable passkey.
  - Use recovery key.
- After unlock, derive and deliver CAK.

### 8.4 Federated SSO Without ZK Client

- User authenticates through upstream IdP.
- DarkAuth maps or creates the local account according to connection policy.
- No ARK unlock is required.
- Issue normal OIDC tokens.

### 8.5 SCIM-Provisioned User First Login

- Admin or external directory provisions the user through SCIM.
- User exists in DarkAuth but may have no unlock method.
- On first successful authentication, if the client requests ZK:
  - Require setup of a DarkAuth password, PRF passkey, recovery key, or trusted-device transfer.
  - Generate ARK client-side.
  - Create initial envelopes.
- If the client does not request ZK, no key setup is required unless policy requires it.

### 8.6 Passkey Login

- If passkey PRF is available and the credential has an ARK envelope, authenticate and unlock in one ceremony.
- If passkey PRF is unavailable, authenticate with passkey and then ask for a separate key unlock method when ZK is requested.
- If a passkey authenticates but cannot unlock, UI copy MUST clearly distinguish "signed in" from "encryption key still locked."

### 8.7 Existing Device Approval

New device:

1. User authenticates by SSO, passkey, passwordless, or password.
2. Browser generates an ephemeral device public key.
3. Browser creates a key approval request on the server.
4. UI asks user to approve from an existing device or enter password.

Existing unlocked device:

1. Receives or polls for approval request.
2. Shows device metadata and a short verification code.
3. User approves.
4. Device unwraps ARK locally.
5. Device encrypts ARK to the new device public key.
6. Device uploads the encrypted approval envelope.

New device:

1. Downloads encrypted approval envelope.
2. Decrypts ARK locally.
3. Creates its own trusted-device envelope if the user chooses to trust it.
4. Derives CAK for the requesting client.

Approval requests MUST expire quickly and be single-use.

### 8.8 Key Setup Journey

If an authenticated user has no ARK and a ZK client requires a key:

1. Explain that encrypted app access requires an encryption unlock method.
2. Ask the user to create at least one primary unlock method:
   - DarkAuth password envelope.
   - PRF-capable passkey envelope when supported.
   - Trusted-device transfer when another unlocked device exists.
3. Generate ARK client-side.
4. Create at least one envelope.
5. Strongly encourage creation of a recovery key.
6. Derive CAK and continue authorization.

Organization policy MAY require a password envelope even when the user authenticated through SSO.

## 9. Federation

DarkAuth supports upstream identity providers as auth methods.

### 9.1 OIDC Federation

Initial federation support SHOULD prioritize upstream OIDC.

Requirements:

- Authorization Code with PKCE for public/browser-facing flows.
- Discovery metadata support.
- JWKS validation.
- Issuer and audience validation.
- Nonce/state validation.
- Configurable claim mapping.
- Account linking policy.
- Domain and organization routing policy.

### 9.2 SAML Enterprise SSO

SAML MAY be implemented after OIDC federation if enterprise demand requires it.

DarkAuth does not implement SAML in this key-management phase. OIDC is the enterprise federation baseline. SAML support requires a separate implementation plan because assertion parsing, XML signature validation, metadata ingestion, ACS routing, and IdP-specific interoperability introduce a distinct security surface from OIDC.

Requirements:

- Signed assertion validation.
- Issuer/entity ID validation.
- ACS URL validation.
- Clock skew limits.
- Claim mapping.
- Account linking policy.

### 9.3 Account Linking

Account linking MUST be explicit and policy-driven.

Supported policies:

- Strict: require existing user link.
- Email-domain: allow auto-link for verified domain and verified email.
- Invite-only: allow link only for invited/provisioned users.
- Admin-only: linking performed by admins.

DarkAuth MUST NOT auto-link solely on unverified email.

### 9.4 Federation Policy

Federation connections MUST define:

- Whether just-in-time user creation is allowed.
- Whether SCIM pre-provisioning is required.
- Whether DarkAuth password setup is required for ZK clients.
- Whether PRF passkey setup is allowed.
- Whether trusted-device approval is allowed.
- Whether users may bypass key setup for non-ZK clients.

Federated logout clears sessions and memory keys. It does not delete envelopes.

## 10. SCIM

SCIM support is for provisioning and lifecycle management.

### 10.1 SCIM Scope

Implement SCIM 2.0 resources:

- `/scim/v2/Users`
- `/scim/v2/Groups`
- `/scim/v2/ServiceProviderConfig`
- `/scim/v2/ResourceTypes`
- `/scim/v2/Schemas`

Support:

- Create user.
- Update user.
- Patch user.
- Deactivate user.
- Create group.
- Update group membership.
- Delete/deactivate group according to policy.
- External ID mapping.

### 10.2 SCIM And Authentication

- SCIM does not log users in.
- SCIM-provisioned users authenticate using DarkAuth password, passkey, or federated SSO.
- SCIM MAY set account state such as active, suspended, department, display name, group membership, and allowed auth policy.
- Deprovisioning MUST revoke sessions and refresh tokens.
- Deprovisioning MUST NOT delete encrypted key material immediately unless configured by retention policy.
- SCIM group changes MUST update DarkAuth organization/group/role mappings before the next token issuance.

### 10.3 Enterprise Policy

Organizations MAY configure:

- Self-registration disabled.
- Only SCIM-provisioned users may sign in.
- Required upstream IdP for a domain.
- Required DarkAuth key unlock for ZK clients.
- Whether users may create password envelopes.
- Whether users may create passkey PRF envelopes.
- Whether trusted-device approval is allowed.

### 10.4 SCIM Mapping

SCIM `User.active=false` maps to a suspended/deprovisioned DarkAuth user. Suspended users MUST NOT complete new login or authorization flows.

SCIM groups map to DarkAuth groups or organization roles according to admin-configured mapping rules. Unknown external groups MAY be ignored, created, or rejected depending on policy.

SCIM delete SHOULD default to deactivate/suspend rather than hard delete, so encrypted data retention can follow organizational policy.

## 11. Passkeys

DarkAuth supports two passkey modes.

### 11.1 Passkey Authentication

Passkeys can authenticate users with WebAuthn.

This does not imply key unlock.

### 11.2 Passkey PRF Unlock

When WebAuthn PRF is supported:

- Registration MUST test that PRF output is returned.
- Authentication MUST request PRF output with the stored salt.
- The returned PRF output derives a wrapping key.
- The wrapping key unwraps ARK from the passkey envelope.

If PRF fails or is unsupported, DarkAuth MUST fall back to another unlock method.

### 11.3 Passkey Registration Outcomes

Passkey registration can produce:

- Auth-only passkey: WebAuthn credential exists, no PRF envelope.
- Auth + unlock passkey: WebAuthn credential exists and a PRF envelope wraps ARK.

The user UI MUST show these differently.

### 11.4 Passkey Portability

Synced passkeys may move between devices. DarkAuth MUST treat each credential according to WebAuthn credential identity and observed PRF behavior, not according to assumptions about the vendor.

If a synced credential returns the same PRF output on multiple devices, the same passkey envelope can unlock ARK. If not, the user needs another unlock method.

## 12. User UI Requirements

The user UI MUST separate these sections:

- Sign-in methods:
  - Password
  - Passkeys
  - Enterprise SSO
  - Connected identities
- Encryption unlock methods:
  - Password envelope
  - PRF passkey envelope
  - Trusted devices
  - Recovery key
- Devices:
  - Current device
  - Trusted devices
  - Pending approvals
  - Revoke device
- Recovery:
  - Create recovery key
  - Rotate recovery key
  - Review recovery status

When a user is authenticated but key material is locked, the UI MUST show that state clearly.

Authorization screen behavior:

- If the client is non-ZK, do not show key unlock controls.
- If the client is ZK and ARK is unlocked, show normal consent/authorization.
- If the client is ZK and ARK is locked, primary action SHOULD be "Accept on another device" when trusted devices exist, with "Enter password" as fallback.
- If no unlock method exists, show key setup.
- If the user is SCIM/SSO managed and policy disallows password envelopes, hide password setup and show allowed methods.

## 13. Admin UI Requirements

Admins need controls for:

- ZK required per client.
- Key delivery version per client.
- Client key migration status.
- Enterprise SSO connections.
- SCIM tokens and mappings.
- Provisioning policies.
- Domain routing.
- Allowed unlock methods.
- Device approval policy.
- User key status without exposing keys.

Admins MUST NOT be able to decrypt user ARKs or CAKs.

Admin visibility:

- Admins may see whether a user has key setup complete.
- Admins may see counts/types of envelopes.
- Admins may revoke envelopes/devices.
- Admins may require reset/setup at next login.
- Admins must not see plaintext keys, PRF outputs, recovery keys, or decrypted envelope contents.

### 14.3 Endpoint Body Sketches

Create envelope:

```json
{
  "key_id": "ark_...",
  "type": "password",
  "label": "Password",
  "wrapping_alg": "OPAQUE-HKDF-SHA256+A256GCM",
  "wrapped_key": "base64url",
  "aad": "base64url",
  "metadata": {
    "version": "v2"
  }
}
```

Create recovery key:

```json
{
  "recovery_key_id": "rk_...",
  "envelope_id": "env_...",
  "key_id": "ark_...",
  "label": "Recovery key",
  "wrapping_alg": "DarkAuth-Recovery-HKDF-SHA256+A256GCM/v2",
  "wrapped_key": "base64url",
  "aad": "base64url",
  "verifier": "base64url 32 bytes",
  "metadata": {
    "version": "v2"
  }
}
```

Rotate account key:

```json
{
  "key_id": "ark_...",
  "version": "v2",
  "retire_old_envelopes": false
}
```

Create device approval:

```json
{
  "new_device_public_jwk": {},
  "client_id": "app-web",
  "state_hash": "base64url",
  "verification_code_hash": "base64url"
}
```

Approve device:

```json
{
  "encrypted_approval": "compact-jwe-or-base64url",
  "approved_device_id": "dev_..."
}
```

Consume device approval:

```json
{
  "request_id": "apr_...",
  "new_device_proof": "base64url"
}
```

Responses MUST avoid returning plaintext keys except to the authenticated browser in encrypted-to-device form.
